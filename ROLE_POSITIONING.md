# Role Positioning

This portfolio is not a claim of external adoption or professional employment
history. It is a map from the public evidence artifacts to concrete technical
functions that teams can evaluate.

## Core Position

I build verification layers around AI systems:

- decision gates before model output becomes action
- evidence packets around benchmark and product claims
- trace/redaction boundaries for reviewer-facing artifacts
- public non-claims that keep scope visible

## Best-Fit Technical Functions

| Function | How the portfolio maps to it | Public evidence |
| --- | --- | --- |
| AI Evaluation | Benchmark runners, challenge packs, conformance checks, failure analysis | Cognitive OS seed/adversarial/challenge runners; X-MoD evidence kit |
| Trust & Safety Engineering | Gate decisions, redaction rules, handoff/deny/degrade behavior, non-claim boundaries | Cognitive OS decision envelopes; Angelos protocol packet |
| LLMOps / Reliability Engineering | Reproduction commands, CI gates, evidence endpoints, issue templates | Cognitive OS verification CI; external reproduction path |
| Guardrails and Policy Systems | Profile-owned policies compiled into executable gate decisions | Cognitive Configuration Profile and gate protocol |
| Evidence-Governed Product Systems | Release gates, claim boundaries, sealed evidence packets | Financial Agent Evidence OS |

## What A Reviewer Can Check

- Clone the repo and run the documented commands.
- Check whether CI passes.
- Inspect whether public evidence exports include raw prompts or candidate text.
- Compare included-suite results with limitation-discovery packs.
- Confirm that non-claims are visible near headline metrics.

## Current Limits

- No external adoption claim.
- No claim of production deployment.
- No claim of broad model safety coverage.
- No claim that the included benchmarks are sufficient for real-world release.
- No claim that CARLA raw workspaces are public reproducibility artifacts.

## Near-Term Proof Targets

1. One outside reproduction report using the documented issue template.
2. One reviewer-provided local benchmark run through the Cognitive OS external adapter.
3. One independent code review issue or pull request.
4. More challenge cases where the system does not score perfectly, with failure notes preserved.

## Reviewer Reading

Read this portfolio as early, evidence-governed engineering work. The important
signal is not popularity. The signal is whether the artifacts are runnable,
measurable, bounded, and honest about what they do not prove.
